# 데이터 / 자격증명 취급 규칙

이 저장소는 사내 보안 로그를 처리하는 에이전트를 만든다. 데이터 취급을 잘못하면 단순한 코드 사고를 넘어 정책 위반이 된다. **아래 규칙은 강제다.**

## Commit 금지

- 실 출입/접근/권한 로그 (`*.xlsx`, `*.xls`, `*.csv`)
- 실 사번, 실명, 부서명, 계정 ID
- SMTP / LLM 서버 / 메일 발송용 자격증명, API 키, 토큰
- 사내 IP, 호스트명, 내부 도메인이 적힌 설정 파일

`.gitignore`에 `data/`, `*.xlsx` 등이 차단되어 있다. 우회해서 강제 add 금지.

## 허용

- `tests/fixtures/` 합성 데이터 (사번/이름은 가짜, 시각은 임의)
- `data/synthetic/` 합성 시나리오 데이터
- `.env.example` (실 값은 비워둘 것)

## 외부 통신 정책

| 무엇이 어디로 | 허용 | 비고 |
|---|---|---|
| 운영 시 실 데이터 → 로컬 LLM (Ollama/vLLM) | ✅ | 사내망 내 |
| 운영 시 실 데이터 → 외부 LLM API (OpenAI/Anthropic 등) | ❌ | 정책 위반 |
| 개발 시 코드 → 외부 코딩 에이전트 (Claude Code 등) | ✅ | 코드만 흐름, 실 데이터는 안 흐름 |
| 메타데이터 (Multica 이슈 제목/설명/코멘트) → Multica 클라우드 | ⚠️ 조건부 | 실 데이터 첨부 금지 |

**핵심 분리 원칙**: 개발 단계에선 외부 API를 통해 *코드*가 흐르지만 *실 데이터*는 흐르지 않는다. 이 분리를 깨면 정책 위반.

## 합성 데이터 작성 규칙

- 결정론적 시드 사용 (`numpy.random.seed(42)` 등)
- 사번은 `EMP-TEST-XXXX` 같이 가짜임이 명확한 prefix
- 이름은 `테스트A`, `Test User 1` 식
- 시간은 합성 기간 내 임의 분포 (실 운영 시간대 답습 금지)
- 실 출입문/시설명 절대 사용 금지 → `Gate-A`, `Building-X` 등 추상명

## 점검 절차

- 새 PR 머지 전: diff에 실 데이터 의심 패턴 검토
  - 7자리 이상 숫자열 (실 사번 가능성)
  - 한국 인명 패턴
  - 사내 도메인 / IP / 호스트명
  - 실 시간 연속 시퀀스 (`2025-01-15 09:23:17` 같은 정밀 timestamp)
- 합성 데이터 생성 PR: fixture 파일 직접 열어서 가짜인지 확인
- `.env` 절대 commit 금지. `.env.example`만 commit, 실 값은 빈칸

## 사고 발생 시

실 데이터가 commit되어 push까지 됐다면:
1. 즉시 보안 담당자에게 보고
2. `git filter-repo` 또는 BFG로 history 정리 (단순 커밋 되돌리기로는 부족)
3. force-push로 원격 history 갱신 (사용자 승인 후)
4. 노출된 자격증명은 즉시 폐기·재발급
